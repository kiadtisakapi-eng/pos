## สิ่งที่ "รันทดสอบจริง" ไม่ใช่แค่อ่านโค้ด

1. **ตรวจ syntax** ทุกไฟล์ JS ด้วย Node → ผ่านหมด
2. **PromptPay payload** — เขียนตัวอ้างอิงอิสระตามมาตรฐาน EMVCo + คำนวณ CRC16 ใหม่เทียบ → ทุกฟิลด์ตรง, CRC ถูกต้องทุกเคส (มือถือ/บัตรปชช./e-Wallet/ใส่ดอกจุด)
3. **QR Code จริงสแกนได้ไหม** — เขียน "ตัวถอดรหัส QR อิสระ" decode ภาพ QR ที่ระบบสร้าง → **คืนค่าเป็น payload เดิมเป๊ะทุกเคส** (version/mask/EC ถูกต้อง) ⇒ QR ที่สร้างเป็น QR มาตรฐานที่อ่านได้จริง
4. **การเงิน end-to-end** — จำลองตะกร้า 2 รายการ + ส่วนลด → ตรวจ subtotal/ส่วนลด/สุทธิ/เงินทอน/ค่าคอม/เงินคาดในลิ้นชัก/แถวที่เขียนลง Sheets → ตรงทั้งหมด
5. **ผูก element id** — เทียบ getElementById 158 จุดในโค้ดกับ id ใน index.html → ครบ ไม่มีหลุด

## ปัญหาที่พบ (เรียงตามความสำคัญ)

### 🟠 ควรตรวจ/ปรับก่อนใช้จริง

**1. PromptPay — ลำดับฟิลด์ต่างจากมาตรฐานที่ทุกเจ้าใช้**
ระบบเรียงฟิลด์เป็น `…29(ร้าน) → 58(ประเทศ) → 53(สกุลเงิน) → 54(จำนวนเงิน) → 63(CRC)`
แต่ไลบรารี/ผู้ให้บริการแทบทุกเจ้าเรียง `53 → 54 → 58` (จำนวนเงิน/สกุลเงิน มาก่อนประเทศ)
- เนื้อหาทุกฟิลด์ถูกต้อง + CRC ถูกต้อง และโดยหลัก EMVCo การอ่านไม่อิงลำดับ ⇒ **โอกาสสูงที่สแกนจ่ายได้ปกติ**
- แต่เพราะเป็นเรื่อง "เงินจริง" และต่างจาก reference ทุกตัว → **แนะนำให้ทดสอบสแกนด้วยแอปธนาคารจริง 1–2 แอป (เช่น KBank, SCB) ก่อนเปิดร้าน** หรือสลับ 2 บรรทัดให้เป็นลำดับมาตรฐาน (แก้ง่ายมาก)
- ไฟล์: `promptpay-qr.js` ฟังก์ชัน `buildPayload`

**2. ความปลอดภัยของ API (รับรู้ไว้)**
`API_SECRET` ฝังอยู่ใน `app.js` ที่ส่งให้เบราว์เซอร์ + GAS ตั้ง "Who has access: Anyone"
⇒ ใครเปิด View Source ก็เห็น secret และยิง API ได้ เป็นข้อจำกัดโดยธรรมชาติของแอปฝั่ง client ล้วน
- สำหรับร้านเล็กถือว่า **รับได้** (กันคนทั่วไป ไม่กันคนตั้งใจ) แต่ควรเข้าใจว่าไม่ใช่การป้องกันที่แท้จริง
- ถ้าต้องการแน่นหนาขึ้น: ผูก deploy กับบัญชี Google เฉพาะ หรือย้าย sync ไป backend ที่ถือ secret เอง

### 🟡 เล็กน้อย (ขัดเงา — ไม่กระทบการใช้งาน)

**3. ลบบิลย้อนหลัง (Void) ไม่รีเฟรชชีตสรุปวัน/เดือน**
ลบแถวบิลใน Sheets ทันที ✅ แต่ KPI ในชีต "สรุป-…" และ "สรุปรายเดือน" ยังเป็นยอดเดิมจนกว่าจะปิดกะหรือกดส่งสรุปใหม่

**4. รายงาน "รายได้แยกบริการ" ใช้ราคาก่อนหักส่วนลด**
ยอดรายได้รวม = ยอดสุทธิ (หลังลด) แต่ตารางแยกตามบริการ = ราคาเต็ม ⇒ เมื่อมีส่วนลด ผลรวมสองส่วนจะต่างกันเท่าส่วนลด (ตัวเลขบิลถูกต้อง เป็นแค่การกระทบยอดในรายงาน)

**5. ใบเสร็จ — ป้าย "รวมทั้งสิ้น" แสดงยอดก่อนหักส่วนลด**
บรรทัด "รวมทั้งสิ้น" โชว์ค่า subtotal (ก่อนลด) ส่วนยอดที่ลูกค้าจ่ายจริงคือ "ราคาสุทธิ" ด้านล่าง (ถูกต้อง) — เมื่อไม่มีส่วนลดไม่เห็นปัญหา แต่ถ้ามีส่วนลดป้ายอาจชวนสับสน ควรเปลี่ยนเป็น "รวมค่าบริการ"

**6. ไม่เก็บ "เงินรับมา/เงินทอน" ลงบิล**
คำนวณเงินทอนถูกต้องตอนจ่าย แต่ไม่บันทึกลง transaction ⇒ ใบเสร็จ/ประวัติไม่โชว์เงินรับ-ทอน (กระทบเฉพาะถ้าต้องการ audit ภายหลัง)

**7. ไอคอน PWA บน iOS**
ใช้ไอคอน SVG (data-URI) — Android ใช้ได้ แต่ iOS บางรุ่นไม่ใช้ SVG เป็นไอคอนหน้าโฮม ควรเพิ่ม `apple-touch-icon` เป็น PNG ถ้าจะติดตั้งบน iPhone/iPad ให้สวย

**8. เอกสารล้าสมัยเล็กน้อย**
`SYSTEM_OVERVIEW.md` เขียน Service Worker เป็น v11 แต่จริงเป็น **v18** — อัปเดตข้อความให้ตรง

## สิ่งที่ตรวจแล้ว "แข็งแรงดี" (ของดีที่มีอยู่)

- **PIN เจ้าของร้าน** แฮช SHA-256 ก่อนเก็บ + auto-migrate จาก plain text เดิม
- **TX ID** ใช้ timestamp + สุ่ม 8 ตัว — กันชนกันหลัง void แล้ว
- **กัน Formula Injection** ใน Sheets (`safeCell` เติม `'` หน้าข้อความที่ขึ้นต้นด้วย =,+,-,@)
- **LockService** ใน GAS กัน race condition ตอนเขียนพร้อมกัน + **Upsert** (บิล ID ซ้ำ = อัปเดตไม่สร้างซ้ำ)
- **CORS pattern ถูกต้อง** — POST แบบ `text/plain` ข้าม preflight, อ่าน response JSON ได้
- **Sync เป็นคิว** — บิลค้างเก็บสถานะ pending, ส่งทีละรายการ + บันทึกหลังส่งสำเร็จ, มี `isSyncing` กันยิงซ้ำ
- **Service Worker** cache ครบ (รวม `promptpay-qr.js` ⇒ QR ทำงาน offline ได้), ข้าม POST, fallback หน้า index ถูกต้อง
- **Validation** เบอร์โทร / กันลูกค้าเบอร์ซ้ำ / ราคา-เวลา > 0
- **Custom modal** (confirm/prompt/toast) แทน dialog เนทีฟที่บล็อกจอแล้ว
- **คำนวณค่าคอม** ผูกกับบริการ (ไม่ใช่พนักงาน) ⇒ เปลี่ยนพนักงานในบิลย้อนหลังแล้วค่าคอมยังถูก
